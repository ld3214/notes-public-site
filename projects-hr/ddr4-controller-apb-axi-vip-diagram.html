APB + AXI VIP 怎么验证 DDR4 Controller

这页讲的是最常见的使用场景：你不直接写 DDR4 pin-level driver，而是用 APB VIP 配寄存器、用 AXI VIP 产生读写流量，再用 monitor、scoreboard、DDR/DFI checker 判断 controller 有没有正确控制 DDR4。

APB VIP 发控制事务 写 timing、地址映射、端口、refresh、DFI/PHY init 相关寄存器；再读状态寄存器确认初始化完成。
AXI VIP 发数据事务 发 write/read burst 到 DDR memory 地址空间，用不同地址、长度、ID、QoS、数据 pattern 制造压力。
Controller 生成 DDR4 行为 把 AXI 请求调度成 ACT、RD、WR、PRE、REF 等 DDR4 操作，并满足 timing 约束。
结果由多层检查判断 协议 checker、read-hash、AXI-HIF、HIF-DFI、DDR model、log postprocess 一起决定 PASS/FAIL。

1. 在这个工程里怎么使用 VIP

APB VIP：作为“软件/寄存器访问通道”

APB VIP 的 master sequencer 会被挂到寄存器模型和 APB sequence 上。测试初始化时，sequence 通过 APB 写 controller 和 PHY 相关 CSR，然后读状态寄存器确认 DDR 初始化、训练、DFI ready。

你的工程里有两种常见用法：一种是通过 RAL/register utilities 间接访问寄存器；另一种是直接跑 dwc_ddrctl_apb_master_wr_sequence / dwc_ddrctl_apb_master_rd_sequence。

APB transaction 的本质
`uvm_do_with(req, {
  xact_type == svt_apb_transaction::WRITE;
  address   == local::address;
  data      == local::data;
})

`uvm_do_with(req, {
  xact_type == svt_apb_transaction::READ;
  address   == local::address;
})
PADDR PWDATA PWRITE PSEL PENABLE PREADY PRDATA PSLVERR

AXI VIP：作为“内存访问 master”

AXI VIP 的每个 master sequencer 对应一个 AXI port。测试可以在所有 port 上并行启动 directed/random AXI sequence，产生 write burst、read burst、不同 ID、不同地址、不同 burst length 的数据流。

典型 directed sequence 会先写一批地址，再读同一批地址。这样可以检查数据是否真的经过 controller 写进 DDR4，又能否正确读回来。

AXI directed sequence 的本质
l_wr_tran.xact_type   = svt_axi_transaction::WRITE;
l_wr_tran.addr        = target_addr;
l_wr_tran.burst_type  = svt_axi_transaction::INCR;
l_wr_tran.burst_length = axi_len;
l_wr_tran.data[k]     = expected_pattern;

execute_xact_and_proc_resp(l_wr_tran, wait_for_rsp);
AWADDR AWLEN WDATA WSTRB BRESP ARADDR RDATA RRESP
你写 test 时主要是在控制 sequence：先初始化子系统，再启动 AXI traffic。你不需要写 AXI pin toggling，也不需要自己写 APB handshake，VIP driver 会把 transaction 转成总线信号。

2. APB 配置到底在配置什么

内存类型和基础模式

设置 DDR4 模式、rank 数、数据宽度、burst 相关选项、是否启用 ECC/DBI/CRC 等。这里决定 controller 后面应该按什么内存规格工作。

MSTR DBICTL ECCCFG

timing 参数

配置 controller 要遵守的 DDR4 timing，例如 activate 到 read/write 的间隔、precharge 时间、refresh 时间、read/write 切换间隔。

tRCD tRP tRAS tFAW tRFC

地址映射

决定 AXI/system address 的哪些 bit 映射到 DDR4 的 rank、bank group、bank、row、column。这个是 DDR controller 验证里非常关键的点。

ADDRMAP rank bank group row column

端口和 QoS

配置 AXI port 是否打开、优先级、读写队列、urgent、high priority read 等。多端口同时访问时，controller 的仲裁行为会被这些配置影响。

refresh 和 low power

DDR4 必须周期性 refresh。controller 还可能进入 self-refresh、power-down 等低功耗状态。APB 配置会决定这些状态机如何启动和退出。

DFI/PHY 初始化

通过 APB 触发 PHY init/training，等待 dfi_init_complete 或相关 status。AXI 流量通常必须等这些 ready 后才能开始。

3. AXI 流量进入 controller 后，DDR4 协议发生了什么

DDR4 内存组织

DDR4 不是一个平坦 SRAM。一个 AXI 地址会被 controller 映射成 DDR4 的 rank / bank group / bank / row / column。

如果目标 row 已经打开，访问可能是 page hit；如果要访问另一个 row，controller 可能需要先 precharge 旧 row，再 activate 新 row，这就是 page miss 或 bank conflict。

地址理解方式
AXI byte address
  -> system address
  -> rank / bank group / bank / row / column
  -> DDR4 command schedule

一次 DDR4 read/write 的典型命令

如果目标 row 还没打开，controller 要先发 ACT 打开 row；等待 tRCD 后才能发 RD 或 WR；完成后可能保持 row 打开，也可能发 PRE 关闭 row。

简化后的 DDR4 命令时序
Write miss:
  ACT  rank/bank/row
  wait tRCD
  WR   column + write data burst
  wait write recovery
  PRE  optional

Read miss:
  ACT  rank/bank/row
  wait tRCD
  RD   column
  wait read latency
  receive read data burst
  PRE  optional

4. 一个典型 APB + AXI 测试怎么跑

1 创建配置 test 随机化或指定 DDR4 配置，例如 rank、bus width、timing、addrmap、AXI port 数。
2 APB 初始化 通过 APB VIP 写 controller/PHY 寄存器，启动 DDR/PHY 初始化和训练。
3 APB polling 读 status register，确认 init done、training done、DFI ready，避免 DDR 未 ready 就发 AXI。
4 AXI 写入 AXI VIP 向 DDR memory 地址写入 pattern。monitor 同时记录写事务。
5 AXI 读回 AXI VIP 从相同或指定地址读回数据，检查 response、last、data。
6 收尾检查 scoreboard 对比数据和内部事务；postprocess 扫 log，找到 completed 且无 error 才 PASS。

5. 验证结果是否正确，是怎么判断的

检查层次

检查层
检查对象
为什么能说明 DDR controller 对或错
APB VIP checker
APB 握手、读写响应、PSLVERR、APB4 sideband。
保证配置事务本身合法，寄存器访问没有总线级错误。
Register predictor / subscriber
APB monitor 观察到的寄存器写入、读回、状态变化。
确认 test 以为写进去的配置，和 monitor 看到的配置一致。
AXI VIP checker
AXI 五通道协议、burst、ID、response、RLAST/WLAST。
保证流量入口合法；如果 AXI 本身错了，不能怪 DDR controller。
Read-hash scoreboard
写入的 byte/pattern 和读回的 byte/pattern。
这是最直观的 end-to-end data integrity：写到 DDR 的数据必须能正确读回来。
AXI-HIF scoreboard
AXI memory transaction 是否正确转换成 controller 内部 HIF 请求。
能抓地址映射、长度、方向、数据、优先级转换中的错误。
HIF-DFI scoreboard
HIF 请求和 DFI/DDR 命令、数据、timing 的对应关系。
能抓 controller 调度 DDR4 命令时的错误，例如 page/bank/timing 行为不符合预测。
DDR/DFI VIP checker
DFI interface、DDR4 command/data/timing。
能抓真正靠近 DDR4 协议侧的 violation。
postprocess
checklog.pl 扫 test.log/compile.log，并要求 COMPLETED - done with Env。
把所有 checker、scoreboard、fatal、timeout 的结果收敛成最终 PASSED/FAILED。

6. 这和“只验证 APB/AXI”有什么不同

如果只是 APB/AXI 协议验证

你只关心 APB 有没有握手成功、AXI 有没有满足五通道协议、response 是否 OKAY。只要总线 transaction 合法，测试可能就通过。

这种测试的重点是 bus protocol，不关心后面的 DDR4 bank、row、timing、refresh。

这里是 DDR controller 验证

APB/AXI 只是入口。真正要验证的是 controller 接到 AXI 请求后，能否正确映射地址、仲裁多端口、安排 DDR4 命令、处理 refresh、满足 timing、保持数据正确。

所以 AXI protocol 全对也不够。只要 controller 把地址映射错、bank 调度错、read data 返回错，scoreboard 仍然会报错。
